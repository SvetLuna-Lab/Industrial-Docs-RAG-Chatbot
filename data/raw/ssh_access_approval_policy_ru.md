# Политика согласования доступа по SSH

Документ-черновик.  
Цель: установить единый порядок выдачи и отзыва доступа по SSH к серверам производственного контура.

## 1. Общие положения

1. Доступ по SSH предоставляется только для выполнения служебных обязанностей.
2. Все запросы на доступ должны проходить формализованную процедуру согласования.
3. Любой доступ по SSH является **персональным** и **именованным**:
   - запрещено использование общих технических учётных записей без ответственного владельца.

## 2. Роли и ответственность

- **Инициатор доступа** — сотрудник, которому необходим доступ для выполнения задач.
- **Руководитель подразделения** — подтверждает деловую необходимость доступа.
- **Сервис ИТ/ИБ** — проверяет обоснованность, риски и применимость политики безопасности.
- **Администратор системы** — фактически создаёт/обновляет учётную запись и ключи доступа.

## 3. Процедура запроса доступа

1. Инициатор оформляет запрос в системе заявок (Service Desk либо аналог):
   - цель доступа (какие задачи выполняются);
   - перечень целевых серверов или групп;
   - тип доступа (чтение логов, развёртывание, администрирование);
   - запрошенный срок действия доступа.
2. Руководитель подразделения утверждает запрос либо отклоняет его.
3. Сервис ИТ/ИБ:
   - проверяет соответствие запросу минимально необходимого доступа;
   - при необходимости предлагает сужение перечня серверов или сроков.
4. После согласования администратор:
   - создаёт/обновляет учётную запись;
   - заносит публичный ключ пользователя;
   - включает пользователя в нужные группы (или роли).

## 4. Срок действия и ревизия доступа

1. Доступ по SSH должен быть **ограничен по времени**:
   - доступ «до отзыва» допускается только для постоянных ролей администрирования.
2. Рекомендуемые сроки:
   - проектные / временные работы — до 3 месяцев;
   - штатные админы — до 12 месяцев с обязательной ежегодной ревизией.
3. Не реже 1 раза в квартал проводится ревизия доступов:
   - выявляются неиспользуемые учётные записи;
   - закрываются доступы уволенных и переведённых сотрудников;
   - уточняется перечень серверов, к которым нужен доступ.

## 5. Запретные практики

Запрещается:

1. Передавать свои SSH-ключи третьим лицам (включая коллег).
2. Использовать общие ключи для нескольких сотрудников.
3. Использовать SSH-доступ для задач, не связанных с производственной деятельностью.
4. Изменять системные настройки без создания записи в заявке / журнале изменений.

## 6. Отзыв доступа

1. Доступ должен быть немедленно отозван в случаях:
   - увольнения или перевода сотрудника;
   - нарушения требований информационной безопасности;
   - выявления фактов компрометации ключа.
2. Сервис ИТ/ИБ обязан:
   - задокументировать причину отзыва;
   - при необходимости инициировать расследование инцидента.

## 7. Логирование и контроль

1. Все SSH-сессии должны логироваться (аудит команд — по возможности).
2. Данные журналов хранятся в соответствии с политикой хранения логов.
3. При расследовании инцидентов логи SSH являются одним из ключевых источников информации.
